# Hosting & Build Layout

This project lives inside the multi-project repo `prettycode/prettycode.github.io`,
which is served by GitHub Pages at `https://prettycode.github.io/`. The
`return-stacked/` folder is one sub-project among several siblings (`csv-editor/`,
`medtrack/`, etc.) and is reachable at:

- **Production:** `https://prettycode.github.io/return-stacked/`
- **Local dev:** `http://localhost:1984/`

## Goals

1. Production URL must be the natural `/return-stacked/` — visitors should not
   have to know about a `dist/` subfolder.
2. Built assets (hashed JS/CSS) must live in `dist/` so the source tree stays
   clean and the diff of each build is small.
3. Local dev must Just Work: `npm run dev`, open `http://localhost:1984/`, no
   path prefixes to remember.
4. The Vite dev entry HTML and the deployed entry HTML must not collide — one
   uses `/src/main.tsx` (only valid under `vite dev`); the other is the built
   output with hashed asset references.

## Layout

| Path | Role |
| --- | --- |
| `return-stacked/dev.html` | Vite dev/build source HTML. Committed. |
| `return-stacked/index.html` | **Build artifact.** Generated by `npm run build`, then committed for deploy. Served at `/return-stacked/`. References `/return-stacked/dist/assets/...`. |
| `return-stacked/dist/` | Hashed JS/CSS plus copied `public/` files (e.g. `favicon.ico`). No HTML — just assets. Committed. |
| `return-stacked/public/favicon.ico` | Source for `dist/favicon.ico` (Vite copies on build). |

Two conceptually distinct base URLs are in play in production:

- **Page base** (`/return-stacked/`) — where the HTML is served.
- **Asset base** (`/return-stacked/dist/`) — where JS/CSS are served.

Vite's `base` config controls the asset base. The page base is enforced by
where `index.html` ends up on disk plus the React Router basename.

## How the pieces fit

### `vite.config.ts`

```ts
export default defineConfig(({ command }) => ({
    base: command === 'build' ? '/return-stacked/dist/' : '/',
    plugins: [
        react(),
        {
            // In dev, `/` should serve dev.html (since index.html is a build
            // artifact and not present during normal dev workflow).
            name: 'serve-dev-html-at-root',
            configureServer(server) {
                server.middlewares.use((req, _res, next) => {
                    if (req.url === '/' || req.url === '/index.html') {
                        req.url = '/dev.html';
                    }
                    next();
                });
            },
        },
    ],
    build: {
        rollupOptions: {
            // Build dev.html (NOT index.html) so that dev.html is the single
            // source of truth for HTML markup.
            input: path.resolve(__dirname, 'dev.html'),
        },
    },
    // ...resolve, css, server unchanged
}));
```

Why conditional `base`:

- In **dev** (`command === 'serve'`), `base: '/'` keeps every URL at the root
  of `localhost:1984`. No `/return-stacked/` prefix locally.
- In **build** (`command === 'build'`), `base: '/return-stacked/dist/'` makes
  Vite emit asset URLs like `/return-stacked/dist/assets/dev-XYZ.js` so they
  resolve correctly when served from GitHub Pages.

### `package.json`

```jsonc
"scripts": {
    "dev": "vite --port 1984",
    "build": "tsc --noEmit && vite build && node -e \"require('fs').renameSync('dist/dev.html','index.html')\""
}
```

The post-build `renameSync` moves the built HTML up one level:
`dist/dev.html` → `index.html` (at the project root). After that:

- `dist/` contains only assets (no HTML).
- `index.html` at the project root is the deployed entry.

### `dev.html` (source)

```html
<link rel="icon" type="image/x-icon" href="%BASE_URL%favicon.ico" />
...
<script type="module" src="/src/main.tsx"></script>
```

`%BASE_URL%` is replaced by Vite at build time with the current `base`:

- Dev: `/favicon.ico` (served from `public/`)
- Prod: `/return-stacked/dist/favicon.ico`

### `src/main.tsx` — router basename

```tsx
<BrowserRouter basename={import.meta.env.PROD ? '/return-stacked' : '/'}>
```

This is the **page** base, not the asset base, so it can't be derived from
`import.meta.env.BASE_URL`. The asset base in prod is `/return-stacked/dist/`,
but the router needs `/return-stacked`.

In dev, `basename: '/'` matches the root URL.
In prod, `basename: '/return-stacked'` matches the deployed page URL.

## Production request flow

1. Browser hits `https://prettycode.github.io/return-stacked/`.
2. GitHub Pages serves `return-stacked/index.html`.
3. Browser fetches `/return-stacked/dist/assets/dev-*.js` and `*.css`.
4. React mounts. Router sees pathname `/return-stacked/`, strips its basename
   `/return-stacked`, matches the `/` route, renders `<Home />`.

## Dev request flow

1. `npm run dev` → Vite at `http://localhost:1984/`.
2. Browser hits `/`. Middleware rewrites to `/dev.html`.
3. Vite serves `dev.html`, transforms `/src/main.tsx` on the fly.
4. Router basename is `/`, routes match at the root.

## Things to remember

- `index.html` at the project root is **generated**. Don't hand-edit it; edit
  `dev.html` and rebuild. Each `npm run build` overwrites it.
- The whole `dist/` folder plus the regenerated `index.html` must be committed
  for the deployed site to pick up changes (GitHub Pages serves directly from
  the default branch of `prettycode.github.io`).
- If you change the deploy URL (e.g., move out of `/return-stacked/`), three
  places must move together:
  1. `vite.config.ts` → `base` for the build branch.
  2. `src/main.tsx` → `BrowserRouter basename`.
  3. The folder name in the parent repo.
- `vite preview` is **not** wired up for this layout — the preview server
  expects HTML inside `outDir`, but we move it out. Verify production builds
  by visiting the deployed URL or by serving the repo root with any static
  server (`npx serve ..`) and visiting `/return-stacked/`.
